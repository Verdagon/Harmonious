# Struct Opacity and Type Dependency Discovery

## The Problem

Consider these toylang definitions:

```
// Toylang source
struct Spaceship<T> {
    Vec<T> wings;
    Engine<T> engine;
}
struct Engine<T> {
    Vec<T> parts;
}

fn buildFleet() -> Vec<Spaceship<RustWing>> {
    let ship = Spaceship<RustWing> {
        wings: Vec<RustWing>::new(),
        engine: Engine<RustWing> { parts: Vec<RustWing>::new() },
    };
    let fleet = Vec<Spaceship<RustWing>>::new();
    fleet.push(ship);
    fleet
}
```

When rustc compiles this, it needs to monomorphize several Rust generic types:
- `Vec<Spaceship<RustWing>>` — the return type
- `Vec<RustWing>` — used inside `Spaceship` and `Engine`
- `Vec<Spaceship<RustWing>>::push` — called in the function body

For `Vec<RustWing>` specifically, rustc needs to generate:
- `layout_of(Vec<RustWing>)` — to know its size
- `drop_in_place::<Vec<RustWing>>()` — to drop it when `Spaceship` is dropped
- `Vec::<RustWing>::push`, `Vec::<RustWing>::new`, etc. — if called by toylang code

The question is: **how does rustc discover that `Vec<RustWing>` needs to exist?**

For toylang *functions*, this is solved cleanly. When `monomorphize_fn("buildFleet", ...)`
fires, the consumer returns:

```rust
MonomorphizeFnResult {
    extern_symbol: "__toylang_impl_buildFleet".to_string(),
    rust_deps: vec![
        (vec_new_def_id, [Spaceship<RustWing>]),   // Vec<Spaceship<RustWing>>::new
        (vec_push_def_id, [Spaceship<RustWing>]),   // Vec<Spaceship<RustWing>>::push
        (vec_new_def_id, [RustWing]),               // Vec<RustWing>::new
    ],
}
```

The library emits phantom `ReifyFnPointer` casts in the MIR stub. Rustc's monomorphizer
sees the casts and stamps out those function instantiations. Discovery happens lazily,
on demand, during monomorphization. The consumer doesn't declare these deps up front.

For toylang *types*, there's no equivalent mechanism. Types don't have MIR bodies. There's
nowhere to put phantom references. And `layout_of` — the only query override that handles
types — returns pure numbers (sizes, offsets, alignment), with no type information that
rustc could follow to discover dependencies.

## The Asymmetry — Concrete Example

Here's how functions and types are currently handled differently:

**Function: `buildFleet`**

```rust
// Stub (what rustc sees):
extern "C" {
    fn __toylang_impl_buildFleet(_deps: *const ()) -> Vec<Spaceship<RustWing>>;
}

// MIR stub (generated by library at monomorphization time):
bb0:
    _3 = const Vec::<Spaceship<RustWing>>::new as fn() -> Vec<...> (ReifyFnPointer)
    _4 = const Vec::<Spaceship<RustWing>>::push as fn(...) (ReifyFnPointer)
    _5 = const Vec::<RustWing>::new as fn() -> Vec<RustWing> (ReifyFnPointer)
    _0 = __toylang_impl_buildFleet(transmute(_3, _4, _5))
    return
```

The function body is opaque. Dependencies discovered lazily. Full consumer control.

**Type: `Spaceship<RustWing>`**

```rust
// Stub (what rustc sees today — exposes internals):
pub struct Spaceship<T> {
    pub wings: Vec<T>,
    pub engine: Engine<T>,
}
```

Rustc can see `Vec<T>` and `Engine<T>` in the field declarations. It discovers type
dependencies through normal type traversal. The consumer must declare its internal
structure up front, before rustc starts.

The desire is to make types as opaque as functions.

## Why Opacity Matters — Concrete Scenarios

**Scenario 1: Rustc enforces field well-formedness**

If the stub declares `pub wings: Vec<T>`, rustc checks that `Vec<T>` is well-formed
wherever `Spaceship<T>` is used. If toylang has a different notion of well-formedness
(e.g. it allows unsized types in positions Rust doesn't), rustc will reject it.

**Scenario 2: Consumer wants to reorder fields**

Toylang might want to lay out `Spaceship<RustWing>` as `[engine, wings]` instead of
`[wings, engine]` for cache performance. If the stub declares fields in order, rustc
may make assumptions about field ordering (even with our `layout_of` override, the
declared order affects `FieldsShape::memory_index`).

**Scenario 3: Consumer adds an internal field**

If toylang adds an internal `refcount: i32` field to `Spaceship`, the stub must be
updated to include it. The stub mirrors the consumer's internal representation.

**Scenario 4: Late generic resolution**

If toylang has conditional types or associated types that aren't resolved until
monomorphization:

```
struct Wrapper<T> {
    T.StorageType storage;   // resolved at monomorphization time
}
```

The stub can't declare `T.StorageType` — it doesn't exist in Rust's type system.
The field type is only known when `monomorphize_type("Wrapper", T=SomeConcreteType)`
fires. With opaque stubs, this works naturally. With declared fields, it can't.

## What `layout_of` Returns

Concretely, for `Spaceship<RustWing>` with two fields:

```rust
LayoutData {
    size: Size::from_bytes(72),          // e.g. two Vecs = 24 + 24 + padding
    align: AbiAndPrefAlign::new(Align::from_bytes(8)),
    fields: FieldsShape::Arbitrary {
        offsets: [Size::from_bytes(0), Size::from_bytes(24)],  // field byte offsets
        memory_index: [0, 1],
    },
    variants: Variants::Single { index: 0 },
    backend_repr: BackendRepr::Memory { sized: true },
    largest_niche: None,
    ...
}
```

Note: **no `Vec<RustWing>` appears anywhere in this output.** It's pure geometry — byte
offsets and sizes. Rustc cannot discover type dependencies from layout data.

## What Rustc Uses Struct Fields For (Beyond Layout)

When rustc parses `struct Spaceship<T> { wings: Vec<T>, engine: Engine<T> }`:

1. **Layout computation** — computes field sizes/offsets from declared types.
   We override this via `layout_of`, so declared fields don't affect layout.

2. **Drop glue generation** — generates `drop_in_place::<Spaceship<RustWing>>()` that does:
   ```
   drop_in_place::<Vec<RustWing>>(&mut self.wings);   // drop field 0
   drop_in_place::<Engine<RustWing>>(&mut self.engine); // drop field 1
   ```
   We override this via `mir_shims`, replacing it with:
   ```
   __toylang_drop_Spaceship(self as *mut ());   // toylang handles its own drops
   ```
   So declared fields don't affect drop behavior.

3. **Monomorphization discovery** — rustc sees `Vec<T>` in the field declarations and knows
   `Vec<RustWing>` must be monomorphized when `Spaceship<RustWing>` is used. **This is the
   one thing we can't override** — there's no query for "what types does this struct
   transitively contain?"

So the only reason stubs need real field types is monomorphization discovery. Everything
else is already overridden.

## Option A: Declare Fields Up Front (Zero-Sized Arrays)

The stubs declare all field types as zero-sized arrays, preserving type information
without affecting layout:

```rust
// Stub for Spaceship<T> — fields are [T; 0] so they're zero-sized
// but rustc still sees the types for monomorphization discovery
pub struct Spaceship<T> {
    wings: [Vec<T>; 0],
    engine: [Engine<T>; 0],
}

// Stub for Engine<T>
pub struct Engine<T> {
    parts: [Vec<T>; 0],
}
```

When rustc encounters `Spaceship<RustWing>`:
- It sees `[Vec<RustWing>; 0]` as a field → discovers `Vec<RustWing>` exists
- It sees `[Engine<RustWing>; 0]` as a field → discovers `Engine<RustWing>` exists
- Recursively, `Engine<RustWing>` has `[Vec<RustWing>; 0]` → already discovered
- All zero-sized, so rustc's own layout computation gives `size = 0`
- Our `layout_of` override fires and returns the real layout (size = 72, etc.)
- Our `mir_shims` override fires for drop glue — calls `__toylang_drop_Spaceship`
- Rustc monomorphizes `Vec<RustWing>` because it saw it in the field declarations

### Concrete stub generation

The consumer's `generate_stubs()` would produce:

```rust
pub struct Spaceship<T> {
    _f0: [Vec<T>; 0],
    _f1: [Engine<T>; 0],
}
pub struct Engine<T> {
    _f0: [Vec<T>; 0],
}
extern "C" {
    pub fn __toylang_impl_buildFleet(_deps: *const ()) -> Vec<Spaceship<RustWing>>;
}
```

### What the consumer must do before calling `run_compiler`

```rust
// The consumer must resolve ALL generic struct instantiations up front.
// For Spaceship<T>, it must know that T can be RustWing, and that
// Vec<T> is a field type. This resolution happens in the toylang
// frontend, before rustc starts.

impl LangCallbacks for Toylang {
    fn generate_stubs(&self) -> String {
        // Must emit [Vec<T>; 0] for each field — requires knowing
        // the field types at stub generation time
        for struct_def in &self.structs {
            for field in &struct_def.fields {
                emit!("_f{}: [{field_type}; 0],");  // must know field_type here
            }
        }
    }
}
```

### Pros
- Simple. Uses rustc's existing monomorphization discovery.
- No changes needed to the library. `MonomorphizeTypeResult::rust_deps` is unnecessary.
- Proven — `[T; 0]` and `PhantomData<T>` are well-established patterns.

### Cons
- **Consumer must know all field types before calling `run_compiler`.** For the example
  above, the toylang frontend must resolve that `Spaceship<T>` contains `Vec<T>` and
  `Engine<T>` before generating stubs. If toylang has complex generics or conditional
  types, this may require running the consumer's own monomorphizer *before* rustc.
- **Stubs expose internal structure.** Rustc sees `[Vec<T>; 0]` and knows the type
  contains a `Vec`. This is less opaque than we'd like.
- **Asymmetric with functions.** Functions get lazy discovery; types don't.

### `MonomorphizeTypeResult::rust_deps`

Not needed. Can be removed from the trait.

## Option B: Opaque Stubs, Lazy Discovery via Functions

The stubs declare types with only `PhantomData` for generic params:

```rust
// Stub for Spaceship<T> — completely opaque, rustc doesn't know the fields
pub struct Spaceship<T>(std::marker::PhantomData<T>);

// Stub for Engine<T> — also opaque
pub struct Engine<T>(std::marker::PhantomData<T>);
```

Rustc knows `Spaceship<T>` is generic over `T` but has no idea it contains `Vec<T>`
or `Engine<T>`. Type dependencies are discovered lazily during function monomorphization.

### Concrete example: how `Vec<RustWing>` gets discovered

1. Rustc's monomorphizer encounters `buildFleet()`.
2. The library's `mir_built` override calls `monomorphize_fn("buildFleet", ...)`.
3. The consumer's monomorphizer walks `buildFleet`'s body:
   - Sees `Spaceship<RustWing> { wings: ..., engine: ... }` — a struct construction
   - Monomorphizes `Spaceship<RustWing>`:
     - Walks fields: `Vec<RustWing>`, `Engine<RustWing>`
     - `Vec<RustWing>` is a Rust type → add to type deps
     - `Engine<RustWing>` is a toylang type → recursively monomorphize
       - Walks fields: `Vec<RustWing>`
       - Already in deps, skip
   - Sees `Vec<RustWing>::new()` — add `Vec::new<RustWing>` to function deps
   - Sees `fleet.push(ship)` — add `Vec::push<Spaceship<RustWing>>` to function deps
4. Consumer returns:
   ```rust
   MonomorphizeFnResult {
       extern_symbol: "__toylang_impl_buildFleet".to_string(),
       rust_deps: vec![
           // Function deps (existing mechanism)
           (vec_new_def_id, [RustWing]),                // Vec<RustWing>::new
           (vec_new_def_id, [Spaceship<RustWing>]),     // Vec<Spaceship<RustWing>>::new
           (vec_push_def_id, [Spaceship<RustWing>]),    // Vec<Spaceship<RustWing>>::push
           // Type deps (new — discovered by walking Spaceship's fields)
           (vec_struct_def_id, [RustWing]),             // Vec<RustWing> the type itself
       ],
   }
   ```
5. Library builds MIR stub with phantom refs for ALL deps:
   ```
   bb0:
       // Phantom function deps (existing ReifyFnPointer mechanism)
       _3 = const Vec::<RustWing>::new as fn() -> Vec<...> (ReifyFnPointer)
       _4 = const Vec::<Spaceship<RustWing>>::new as fn() -> Vec<...> (ReifyFnPointer)
       _5 = const Vec::<Spaceship<RustWing>>::push as fn(...) (ReifyFnPointer)

       // Phantom type deps (new mechanism — zero-sized locals)
       StorageLive(_6)     // _6: Vec<RustWing>  — forces rustc to know this type exists
       StorageDead(_6)

       // Real call to the consumer's compiled function
       _0 = __toylang_impl_buildFleet(transmute(_3), transmute(_4), transmute(_5))
       return
   ```
6. Rustc's monomorphizer sees `Vec<RustWing>` in the local declarations → monomorphizes it
   (generates drop glue, layout, etc.).

### What the consumer's `monomorphize_fn` does in detail

```rust
impl LangCallbacks for Toylang {
    fn monomorphize_fn(&self, name: &str, tcx: TyCtxt, def_id: LocalDefId)
        -> MonomorphizeFnResult
    {
        let func = self.registry.get(name);
        let body = func.body.unwrap();

        let mut fn_deps = vec![];
        let mut type_deps = vec![];

        // Walk the function body, discovering deps
        self.walk_body(body, tcx, &mut fn_deps, &mut type_deps);

        // Combine both kinds of deps into rust_deps
        let mut rust_deps = fn_deps;
        rust_deps.extend(type_deps);

        MonomorphizeFnResult {
            extern_symbol: format!("__toylang_impl_{}", name),
            rust_deps,
        }
    }
}

// The consumer's monomorphizer — walks bodies AND struct fields
impl Toylang {
    fn walk_body(&self, body: &Body, tcx: TyCtxt,
                 fn_deps: &mut Vec<...>, type_deps: &mut Vec<...>) {
        for stmt in &body.stmts {
            match stmt {
                // Found a struct construction: Spaceship<RustWing> { ... }
                StructLit { type_name: "Spaceship", type_args: [RustWing] } => {
                    // Recursively walk the struct's fields to find Rust type deps
                    self.walk_struct_fields("Spaceship", [RustWing], tcx, type_deps);
                }
                // Found a Rust function call: Vec::new()
                StaticCall { ty: "Vec", method: "new" } => {
                    let def_id = find_vec_method(tcx, "new");
                    fn_deps.push((def_id, tcx.mk_args(&[...])));
                }
                // ... etc
            }
        }
    }

    fn walk_struct_fields(&self, name: &str, type_args: &[Ty], tcx: TyCtxt,
                          type_deps: &mut Vec<...>) {
        let struct_def = self.registry.structs.get(name);
        for field in &struct_def.fields {
            let concrete_ty = substitute(field.ty, type_args);
            if is_rust_type(concrete_ty) {
                // e.g. Vec<RustWing> — add as a type dep
                type_deps.push((rust_def_id, rust_generic_args));
            } else if is_toylang_type(concrete_ty) {
                // e.g. Engine<RustWing> — recursively walk
                self.walk_struct_fields("Engine", [RustWing], tcx, type_deps);
            }
        }
    }
}
```

### How the library distinguishes function deps from type deps

`MonomorphizeFnResult::rust_deps` is `Vec<(DefId, GenericArgsRef)>`. The library checks
each `DefId` to determine what kind of phantom reference to emit:

```rust
for (def_id, args) in &result.rust_deps {
    match tcx.def_kind(def_id) {
        DefKind::Fn | DefKind::AssocFn => {
            // Emit ReifyFnPointer cast (existing mechanism)
            emit_phantom_fn_cast(def_id, args);
        }
        DefKind::Struct | DefKind::Enum => {
            // Emit zero-sized local declaration (new mechanism)
            emit_phantom_type_local(def_id, args);
        }
        _ => { /* ignore or panic */ }
    }
}
```

### Pros
- **Symmetric with functions.** Both types and functions have lazy dependency discovery.
- **Stubs are fully opaque.** Rustc doesn't see internal field structure.
- **Consumer has maximum flexibility.** Can change field layout, field ordering, add
  internal fields — without changing stubs.
- **Consumer doesn't need to resolve generics up front.** Dependencies discovered on
  demand during monomorphization.
- **Handles conditional/associated types.** If a field type is `T.StorageType`, the
  consumer resolves it at monomorphization time and returns the concrete type as a dep.

### Cons
- **Every Rust type dep must be reachable through some function.** If `Spaceship<RustWing>`
  contains `Vec<RustWing>` but no function ever constructs a `Spaceship<RustWing>`, the
  dep won't be discovered. In practice: structs must be constructed somewhere, so this
  can't happen.
- **More complex library implementation.** The MIR builder needs to emit phantom locals
  for type deps alongside phantom casts for function deps.
- **Unverified.** The phantom local approach for type monomorphization needs testing —
  does rustc's monomorphizer actually follow types in unused local declarations?
  If not, we may need a different mechanism (e.g. `let _: MaybeUninit<Vec<RustWing>>`
  or a dummy allocation).
- **Consumer's monomorphizer must walk struct fields.** The consumer can't just look at
  the function body — it must transitively walk any toylang struct it encounters to find
  Rust types inside. This is extra work for the consumer, but it's work the consumer's
  monomorphizer would do anyway.

### `MonomorphizeTypeResult::rust_deps`

Not needed in Option B. All type deps flow through `monomorphize_fn`. The consumer walks
struct fields during function monomorphization and returns everything in one place.

`MonomorphizeTypeResult` simplifies to just `field_types` (for layout computation).

## Phantom Type Local: Verification and Fallback

### Primary approach: unused local declaration

The simplest way to emit a type dep is an unused local in MIR:

```
StorageLive(_6)     // _6: Vec<RustWing>
StorageDead(_6)
```

**Unverified.** It's unclear whether rustc's monomorphization collector follows types
in local declarations that are never actually read or written. The collector walks
reachable basic blocks looking for `Rvalue`, `Operand`, and `Terminator` references.
A `StorageLive`/`StorageDead` pair doesn't contain an `Rvalue` or `Operand` — just a
local index. The local's *type* is in `local_decls`, but the collector may not scan
`local_decls` types for unused locals.

Needs testing. If it works, this is the preferred approach (zero cost, no runtime effect).

### Fallback: make the type "used" by feeding it into the extern call

If unused declarations don't trigger monomorphization, make the type's value flow into
the extern call so the collector can't ignore it:

```
bb0:
    // Create a zero-sized array of the type
    StorageLive(_6)                        // _6: [Vec<RustWing>; 0]
    _6 = []                                // zero-sized, no actual allocation

    // Call size_of on it (always returns 0)
    _7 = std::mem::size_of_val(&_6)        // _7: usize = 0

    // Add to a running sum (all zeros, doesn't affect the real call)
    _8 = Add(_phantom_sum, _7)             // still 0

    StorageDead(_6)

    // ... later, pass _phantom_sum as an extra arg to the extern call
    _0 = __toylang_impl_buildFleet(..., _phantom_sum)
```

Because `_6` is read by `size_of_val`, and the result flows into the extern call,
the collector must treat `[Vec<RustWing>; 0]` as "used." The value is always 0 so
it has no runtime effect on the call.

This is more complex MIR but guaranteed to work — the type is transitively used by
a reachable call, which is the strongest monomorphization guarantee.

### `mentioned_items` — rejected

See `docs/historical/design-monomorphization-triggers.md` for full analysis. Key problems:
- Mentioned items go into `state.mentioned`, not `state.visited` — NOT guaranteed to be
  codegen'd.
- The docs say: "this is *not* soundness-critical and the contents of this list are *not*
  a stable guarantee."
- Lifecycle conflict with `mir_promoted` which also sets `mentioned_items`.
- Outside the intended purpose (designed for const-eval error surfacing).

The phantom `ReifyFnPointer` approach (for function deps) and phantom local/used-value
approach (for type deps) both put items in `state.visited`, which is the guaranteed path.

## Conclusions

1. **`MonomorphizeTypeResult::rust_deps` should be removed.** In both options, type deps
   are either discovered by rustc from stub fields (Option A) or by the consumer's
   monomorphizer and returned through `monomorphize_fn` (Option B). `monomorphize_type`
   only needs to return `field_types` for layout computation.

2. **The consumer's `monomorphize_fn` is the single discovery point for ALL deps** in
   Option B — both function deps (Rust generic function instantiations) and type deps
   (Rust generic type instantiations). The consumer walks function bodies AND struct
   fields transitively.

3. **`mentioned_items` is rejected** for both function and type deps. The phantom
   approach (ReifyFnPointer for functions, locals for types) provides stronger guarantees.

4. **Dead code to clean up:** `build_const_i32_body` (never called),
   `GLOBALIZE_SYMBOLS` (set but never read).

---

## Current State

- `MonomorphizeTypeResult::rust_deps` exists in the trait but is always `vec![]` and
  nothing in the library reads it. Should be removed.
- Stubs currently declare real field types (accurate, not zero-sized). This works because
  toylang's types only have primitive fields (`i32`, `i64`, etc.) — no Rust generics yet.
- The library only emits phantom `ReifyFnPointer` casts for function deps. No mechanism
  exists yet for type deps.
- `build_const_i32_body` in mir_helpers is dead code (never called).
- `GLOBALIZE_SYMBOLS` in codegen_wrapper is dead code (set but never read).

## Decision

**Confirmed: Option B** (opaque stubs, lazy discovery via `monomorphize_fn`).

Key risk: the phantom local approach for type monomorphization is unverified (with a
fallback if it doesn't work — see "Phantom Type Local: Verification and Fallback" above).

**Additional decision: accessor methods for Rust-side field access.**

Because stubs are opaque, Rust code cannot access toylang struct fields directly (e.g.
`ship.wings` won't compile). Instead, toylangc generates `impl` blocks with accessor
methods in the stubs:

```rust
pub struct ToyShip(std::marker::PhantomData<()>);

impl ToyShip {
    pub fn wings(&self) -> &Vec<i32> {
        extern "C" { fn __toylang_accessor_ToyShip_wings(s: &ToyShip) -> &Vec<i32>; }
        unsafe { __toylang_accessor_ToyShip_wings(self) }
    }
}
```

The accessor body (a GEP to the field's byte offset) is compiled by toylangc's LLVM
backend into the .o file. Toylang itself stays UFCS with no `impl` blocks — the
`impl` blocks in stubs are purely a Rust-facing presentation layer.

This design extends naturally to trait conformance: `impl Trait for ToyStruct` blocks
would similarly delegate to toylang top-level functions via extern symbols. toylangc
controls how Rust sees each function via a `RustPresentation` enum:
- `FreeFunction` — top-level `pub fn`
- `Method { on_type }` — `impl Type { pub fn ... }`
- `TraitMethod { on_type, trait_name }` — `impl Trait for Type { fn ... }`

Implementation steps for Option B:
1. Test that phantom locals trigger monomorphization in rustc
2. If not, implement the "used value" fallback (size_of_val + sum into extern call)
3. Change stub generation to emit opaque structs (`PhantomData<T>` only)
4. Generate accessor method `impl` blocks in stubs
5. Compile accessor bodies (GEP) in LLVM backend
6. Update `monomorphize_fn` to walk struct fields and return type deps
7. Update MIR builder to emit phantom locals (or used-value fallback) for type deps
8. Remove `MonomorphizeTypeResult::rust_deps`
9. Update all tests from direct field access (`c.value`) to accessor methods (`c.value()`)
10. Clean up dead code (`build_const_i32_body`, `GLOBALIZE_SYMBOLS`)
